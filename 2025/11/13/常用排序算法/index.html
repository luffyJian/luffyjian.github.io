十大排序算法

选择排序

升序排序最直接的思路：

  1. 从序列中找到最小的元素，将它交换到序列的第一个位置。
  2. 从序列中剩下的元素中找到最小的元素，将它交换到序列的第二个位置。
  3. 重复以上步骤，直到序列排序完成。

以上算法就叫选择排序。

选择排序的时间复杂度是$O(n^2)$，空间复杂度是$O(1)$，是一种原地排序算法。

因为每次交换最小元素和当前元素操作，选择排序是一种不稳定的排序算法。

冒泡排序

算法思路：

  1. 从序列的第一个元素开始，比较相邻的两个元素，如果第一个比第二个大，就交换它们的位置。
  2. 对每一对相邻的元素做同样的工作，从开始第一对到结尾的最后一对，最后的元素会是最大的数。
  3. 针对所有的元素重复以上的步骤，除了最后一个。
  4. 重复以上步骤，直到序列排序完成。

冒泡排序的时间复杂度是$O(n^2)$，空间复杂度是$O(1)$，是一种原地排序算法。

因为每次交换的是相邻逆序对的位置，故冒泡排序是一种稳定的排序算法。

插入排序

算法思路：

  1. 从序列的第二个元素开始，将它插入到前面已经排序好的序列中。
  2. 重复以上步骤，直到序列排序完成。

插入排序的时间复杂度是$O(n^2)$，空间复杂度是$O(1)$，是一种原地排序算法。

因为每次插入操作都保证了前面的元素是有序的，故插入排序是一种稳定的排序算法。

在最好情况下，序列已经有序，插入排序的时间复杂度是$O(n)$。初始有序度越高，插入排序的效率就越高。

希尔排序

h 有序数组：一个数组是 h 有序的，是指数组中任意间隔为 h 的元素都是有序的。

当一个数组完成排序时，其实就是 1 有序的。

算法思路：

  1. 选择一个增量序列$h_1, h_2, …, h_k$，其中$h_i > h_j$，$k$是排序轮数。
  2. 对序列进行$k$轮排序，每轮排序根据增量序列进行插入排序。
  3. 最后一轮排序增量为$1$，对整个序列进行插入排序。

下面选择 h 增量函数为 $2^{k - 1}$

希尔排序的时间复杂度是小于$O(n^2)$，空间复杂度是$O(1)$，是一种原地排序算法。

当 h 大于 1 进行排序时，就可能打乱相同元素的相对位置了，故希尔排序是一种不稳定的排序算法。

在最好情况下，序列已经有序，希尔排序的时间复杂度是$O(n)$。初始有序度越高，希尔排序的效率就越高。

快速排序

算法思路：

  1. 选择一个基准元素，将序列分为两部分，一部分比基准元素小，一部分比基准元素大。这样基准元素就被放到了它最终的位置上。
  2. 对两部分分别递归进行快速排序。

快速排序时间复杂度是$O(nlogn)$，空间复杂度是$O(logn)$，是一种不稳定的排序算法。

处理边界细节的技巧就是，你需要明确每个变量的定义以及区间的开闭情况。

归并排序

算法思路：

  1. 把长度为n的输入序列分成两个长度为n/2的子序列。
  2. 对这两个子序列分别采用归并排序。
  3. 将两个排序好的子序列合并成一个最终的排序序列。

归并排序时间复杂度是$O(nlogn)$，空间复杂度是$O(n)$（需要额外的数组空间辅助），是一种稳定的排序算法，取决于merge 函数的实现。

堆排序

堆排序的两个关键步骤：

  1. 原地建堆：直接将待排序的数组原地变成一个二叉堆
  2. 排序：将元素不断从堆中取出，最终得到有序的结果

这里面一个关键点是要用大顶堆来完成 nums 从小到大的排序，因为从堆顶删除的元素是从后往前填到 nums 数组中的，最终 nums 中的元素是从小到大排序的。

如果你用小顶堆的话，最终 nums 中的元素是从大到小排序的，还需要再翻转一下数组，没有大顶堆的效率高。

分析上面堆排序时间复杂度，假设 nums 的元素个数为 N:

  • 第一步建堆过程，swim 方法的时间复杂度是 $O(log(N))$，算法对每个元素调用一次 swim 方法，所以总时间复杂度是 $O(Nlog(N))$。
  • 第二步排序过程，sink 方法的时间复杂度是 $O(log(N))$，算法对每个元素调用一次 sink 方法，所以总时间复杂度是 $O(Nlog(N))$。

综上，堆排序的时间复杂度是 $O(Nlog(N))$。

上述算法直接对 nums 数组进行排序，没有使用额外的空间，所以空间复杂度是 $O(1)$。

堆排序算法是一种不稳定的排序算法，因为二叉堆本质上是把数组抽象成了二叉树的结构，在二叉树逻辑结构上的元素交换操作映射回数组上，无法顾及相同元素的相对位置。

利用逆向思维进行优化

首先，我们只关注二叉堆的逻辑结构，完全二叉树。
在上面解法中，对每个元素依次调用 swim 方法建堆，相当于把数组中的第一个元素作为堆顶（完全二叉树的根节点），然后不断把新元素追加到堆底（完全二叉树的最底层最右侧），调用 swim 方法让新元素上浮到合适的位置。

那么，在这个过程，其实是没有任何优化空间的，因为每个新元素都必须加到堆里，且必须进行上浮操作，才能维护堆的性质。

想要找到优化空间，需要利用二叉堆的一个性质：对于一个二叉堆来说，其左右子堆（子树）也是一个二叉堆。

如果给我一个二叉堆节点，和两个二叉堆，那么可以把这个节点作为堆顶（根节点），两个二叉堆作为左右子堆，构建一个新的二叉堆（只需对根节点进行一次 sink 下沉操作）。

计数排序

计数排序原理：统计每种元素出现的次数，进而推算出每个元素在排序后数组中的索引位置，最终完成排序。

计数排序的时间复杂度和空间复杂度都是 $O(N + max - min)$，其中 $N$ 是数组的长度，$max$ 是数组中的最大值，$min$ 是数组中的最小值， $max - min$ 是数组中的元素范围。

处理负数和自定义类型时，需要有一个映射函数，将元素映射到非负整数索引上。计数排序的映射函数的关键指标是不能丧失原数据的大小关系。

算法思路：

  1. 先遍历一遍数组，找到数组中的最大值 max，最小值 min。
  2. 创建一个长度为 max - min + 1 的计数数组 count，用于统计每个元素出现的次数。
  3. 再次遍历数组，将每个元素 nums[i] 映射到 count 数组的索引 nums[i] - min 上，将对应位置的元素加一。
  4. 最后，从 count 数组的第一个元素开始，将每个元素出现的次数累加起来，就可以得到每个元素的排序位置。
  5. 最后，根据排序位置，将元素放到原数组中对应的位置上。

倒序遍历保证计数排序的稳定性。

因为累加 count 数组后，其中存储的是排序后元素的末尾索引，所以 sorted[count[nums[i] + offset] - 1] = nums[i]; 也是从末尾开始向前填充。

如果倒序遍历 nums 数组，就能保证稳定性，即相同元素的相对顺序不会改变。

count 数组的大小是 max - min + 1，和数组元素的绝对大小无关，只和 max - min 有关，即元素范围有关。

计数排序不是原地排序，需要 count 数组辅助，消耗 $O(max - min)$ 的空间，另外它不能直接把排序结果写到 nums 数组中，需要额外的 sorted 数组来存储排序结果，消耗 $O(N)$ 的空间。

综上，计数排序的空间复杂度是 $O(max - min + N)$，其中 $N$ 是数组的长度，$max$ 是数组中的最大值，$min$ 是数组中的最小值， $max - min$ 是数组中的元素范围。

根据这个排序的特点，当待排序数组中的极值范围跨度很大时，计数排序的空间复杂度会很大，不是明智选择。

代码中有几个单层的循环，分别遍历 nums 数组、count 数组，所以总的时间复杂度是 $O(N + max - min)$，其中 $N$ 是数组的长度，$max$ 是数组中的最大值，$min$ 是数组中的最小值， $max - min$ 是数组中的元素范围。

非比较排序：
从字面上理解，即代码中不包含 if(nums[i] > nums[j]) 这样的比较逻辑的排序算法。

之所以能不使用比较逻辑，本质在于将原数据映射到一个自带有序性的参考系中（比如数组索引），然后借助这个参考系推导出排序后的结果。

反之，如果待排序的数据不能找到这样一个映射和参考系，那么就不能使用非比较排序。

非比较排序的效率一般要高于通用的比较排序，在特定场景下的时间复杂度是线性的。

桶排序

桶排序分析

桶排序算法的核心思想分三步：

  1. 将待排序元素使用映射函数分配到若干个桶中。
  2. 对每个桶中的元素进行排序。
  3. 最后将这些排好序的桶进行合并，得到排序结果

有点像归并排序，都是把大的数组分成小的数组进行排序，最后再合并起来。

不过桶排序更加灵活，三个核心步骤的每一步都能变化：

  1. 如何将待排元素分配到桶中？你需要决定桶的数量，并提供一个映射函数。
  2. 如何对每个同种的元素排序？理论上可以使用任意排序算法，或者模拟归并排序思路，对每个桶递归的运行桶排序。
  3. 如何将排好序的桶合并起来？且时间复杂度为 $O(N)$？

先探讨第二个问题，有没有思考过，为什么要把待排序数组分成若干个桶，然后对每个桶进行排序？这样排序和直接对整个待排序数组相比，有区别吗？

答案是，如果暂时不考虑合并有序桶的算法复杂度，那么分开排序当然要比整体排序效率高。

以最简单的选择排序为例，直接对整个数组排序，那么时间复杂度为 $O(N^2)$，其中 $N$ 是数组的长度。

而如果将数组分成 $k$ 个桶，对每个桶使用选择排序，那么总的时间复杂度是大于 $O(N^2)$，还是小于 $O(N^2)$？

转化成数学问题就是，一个正整数 $n$， 可以分解为 $n = n_1 + n_2 + … + n_k$，那么 $n^2$ 和 $n_1^2 + n_2^2 + … + n_k^2$ 谁大？

答案是 $n^2$ 大于 $n_1^2 + n_2^2 + … + n_k^2$。

当 $k$ 无限大，即 $n_1, n_2, … ,n_k$ 都无限小，那么 $n^2$ 和 $n_1^2 + n_2^2 + … + n_k^2$ 就会无限接近 $n$。

以此观之，如果桶排序将待排元素分配到尽可能多的桶中（ $k$ 尽可能大），即每个桶至多包含一个元素，那么桶排序就转化成了计数排序，其时间复杂度也降到了 $O(N)$。

即便不能做到每个桶只有一个元素，只要 $k > 1$，那么桶排序的时间复杂度就会小于 $O(N^2)$。 $k$ 越大，时间复杂度就越接近 $O(N)$。

反之，如果取最小值 $k = 1$，那么桶排序就退化成了选择排序，时间复杂度就会等于 $O(N^2)$。

桶排序的实现

桶排序可以自选对每个桶的排序算法，一般选择插入排序，因为插入排序属于比较简单的稳定排序算法，且在几个 $O(N^2)$ 基本排序算法中，插入排序的综合能力较强。

还有一种做法就是，对每个桶排序递归地使用桶排序，最终也能把所有元素排好序，这个算法的问题就是会消耗较多的空间复杂度，因为要递归建桶，同一个元素可能存储到了多个桶中，相当于重复存储，会消耗额外的存储空间。

将元素分配到 $k$ 个桶

桶的数量 $k$ 会影响算法的时间复杂度，所以可以把桶数量 $k$ 作为一个参数传入。

如何将待排序数组分配到 $k$ 个桶中呢？

直接求模？不行。 这里选择的映射函数，直接决定了合并有序桶的时间复杂度。，如果使用求模这种方式映射，那么合并有序桶的时间复杂度会超过 $O(N)$。

如下面例子，假设现在有待排序数组 nums 中有 0~99 这一百个数字，乱序排列，现在需要分配到 $k = 10$ 个桶中，有下面两种分配方式：

  • 第一种，求模映射，nums[i] % 10 == 0 的元素都分配到第一个桶，nums[i] % 10 == 1 的元素都分配到第二个桶，以此类推，nums[i] % 10 == 9 的元素都分配到第十个桶。

  • 第二种，向下取整映射，nums[i] / 10 == 0 的元素都分配到第一个桶，nums[i] / 10 == 1 的元素都分配到第二个桶，以此类推，nums[i] / 10 == 9 的元素都分配到第十个桶。

对于第一种方法，第一个桶排好序后结果为 0, 10, 20, ..., 90，第二个桶排好序后结果为 1, 11, 21, ..., 91，以此类推，第十个桶排好序后结果为 9, 19, 29, ..., 99。

合并这 $k$ 个桶就只能借助二叉堆来合并，时间复杂度为 $O(Nlogk)$。

对于第二种方法，第一个桶排好序后结果为 0, 1, 2, ..., 9，第二个桶排好序后结果为 10, 11, 12, ..., 19，以此类推，第十个桶排好序后结果为 90, 91, 92, ..., 99。

合并这 $k$ 个桶的时间复杂度为 $O(N)$，因为每个桶中的元素都是有序的，所以只需要依次连接合并即可。

桶排序的稳定性

桶排序的稳定性主要取决于对于每个桶的排序算法，上面给出的两种桶排序都是稳定的。

首先，分配元素到 $k$ 个桶的过程，是按顺序遍历 nums 数组的，所有相同的元素必然会被分配到同一个桶中，且相对顺序不会改变。

最后合并多个桶的过程，是按顺序遍历桶中的元素，所以稳定性主要取决于对每个桶的排序算法。

如果对每个桶中的元素使用插入排序，因为插入排序是稳定排序，所以排序过程中相同元素的相对顺序也不会改变。因此整个桶排序算法是稳定的。

如果对当前桶递归地使用桶排序，那么当前桶的元素分配到新的桶中时，相同元素的相对顺序也不会改变，以此类推，最终的排序结果也没有改变相同元素的相对位置，所以排序结果也是稳定的。

如果你使用了其他排序算法，比如选择排序，那么桶排序的稳定性就无法保证了。

基数排序（Radix Sort）

基数排序是计数排序 算法的扩展，它主要思路是对待排序的每一位以此进行计数排序。由于计数排序是稳定的，所以对每一位完成计数排序后，所有元素就完成了排序。

基数排序，基数（Radix）其实就是进制的意思，比如十进制数的基数就是10，二进制数的基数就是2。看这个名字就知道这个排序算法肯定和数字的进制有关，进而可推断，这个算法不是通用的排序算法，待排序数据必须是整数，或者能够通过某种规则转化成整数，才能使用基数排序。

为什么每一位必须使用稳定排序

举例

个位数已经排好序，现在要对十位数排序，十位数都是5，稳定排序可以保证这两个5的顺序不变，最终的结果就是正确的；而如果使用不稳定排序，这两5的顺序就可能被打乱，最终结果就会错误。

使用什么稳定排序比较好

最好使用计数排序，因为它的时间复杂度是 $O(n + k)$，其中 $n$ 是数组的长度，$k$ 是数组中元素的取值范围。主要取决于待排序元素的范围，而对基数排序的场景，十进制数的每一位都是0-9，所以计数排序的时间空间复杂度都是 $O(n + 10) = O(n)$，是一个常数级别的时间空间复杂度，所以计数排序是最好的选择。

位数不同怎么办

如果待排序数组中的数字的位数不同，可以在逻辑上把所有数字都补齐成相同的位数，位数不足在高位补0。

有负数怎么办

这个问题有多种解决办法，比如把证书和负数分开排序，最后再合并。

不过最简单的方法是模仿计数排序，把所有数字都加上一个偏移量，使所有数字都变成非负数，排序完成后再减去偏移量，就可以得到正确的排序结果。

必须从低位到高位吗（LSD和MSD）

对于普通的整数排序，必须按照从低位到高位的顺序进行排序，举个例子

如果按照从高位到低位排序，最后对个位数排序，会认为90比18小，这显然不对。

基数排序分为从高位到低位（Most Significant Digit first，简称MSD）和从低位到高位（Least Significant Digit first，简称LSD）两种。

上面的整数排序算法使用LSD，MSD的主要应用场景是字典序排序，其排序算法需要用到递归，和LSD有较大差异。

假设有一个正整数 num，从右往左数（LSD），想要获取第 k 位数字（k 从0开始），可以使用 $(nums / 10^k) % 10$ 这个公式。

即使 num 的总位数小于 k，则 $(nums / 10^k) % 10 == 0$，也是正确的

时空复杂度及稳定性

对于LSD基数排序，由于对每一位的排序都是稳定的，所以最终的排序结果也是稳定的。

假设待排序数组长度为 n，最大元素的位数为 m，LSD基数排序本质上执行力m次计数排序。

前文分析，计数排序的时空复杂度都是 $O(n + max - min)$，在十进制整数的计数排序的场景中，$max - min$ 的值是常数10，可以忽略，所以每次基数排序的时空复杂度都是 $O(n)$。

因此，LSD基数排序的时间复杂度是 $O(mn)$，空间复杂度是 $O(n)$。
