哈希表核心原理

哈希表的基本原理

哈希表可以理解为一个加强版的数组。

数组可以通过索引在 $O(1)$ 的时间复杂度内查找对应的元素，索引是一个非负整数。

哈希表是类似的，可以通过 key 在 $O(1)$ 的时间复杂度内查找这个 key 对应的 value。 key 的类型可以是数字、字符串等多种类型。

怎么做的？特别简单，哈希表的底层实现就是一个数组（可称之为 table）。它把这个 key 通过一个哈希函数（称之为 hash）转化成数组里面的索引，然后增删改操作和数组基本一致。

几个关键概念及原理

key 是唯一的，value 可以重复

将 key 类比数组索引理解。不可能出现两个相同的 key，而 value 是可以重复的。

哈希函数

哈希函数的作用是，把任意长度的输入（key）转化成固定长度的输出（索引）。

增删改查都会用到哈希函数来计算索引值，这个函数的性能很关键。

哈希函数还需要保证，输入相同的 key，输出也必须相同，这样才能保证哈希表的正确性。

哈希冲突

如果两个不同的 key 通过哈希函数得到了相同的索引，这种情况就叫哈希冲突。

哈希冲突不可避免，只能在算法层面妥善处理出现哈希冲突的情况。

哈希冲突是一定会出现的，因为这个 hash 函数相当于将一个无穷大的空间映射到了一个有限的索引空间，所以必然有不同的 key 映射到同一索引上。

两种常见解决哈希冲突的方法，一种是拉链法，另一种是线性探测法（也常被叫做开放寻址法）。

拉链法相当于哈希表底层数组并不是直接存储 value 类型，而是存储一个链表，当有多个不同的 key 映射到同一个索引上，这些 key -> value 对 就存储在这个链表中，这样就能解决哈希冲突的问题。

而线性探测法思路是，一个 key 发现算出来的 index 值被别人占用了，那么它就去 index + 1 的位置看看，如果还是被占用，则继续往后找，直到找到一个空位。

扩容和负载因子

拉链法和线性探测法虽然能解决哈希冲突，但是它们会导致性能下降。

拉链法，需要后续遍历链表找到对应 value 值。这个过程时间复杂度为 $O(K)$，k 是这个链表的长度。

线性探测法也类似，你需要查看对应索引位置是否是需要的值，且并不能确定这个 key 真的不存在，必须顺着索引往后找，直到找到一个空的位置或者找到这个 key 为止，这个过程也是 $O(K)$，k 为连续探查的次数。

所以，如果频繁出现哈希冲突，那么 k 值就会增大，这个哈希表的性能就会下降，这是我们需要避免的。

为什么会出现频繁的哈希冲突？

  1. 哈希函数设计的不好，导致 key 的哈希值分布不均匀，很多 key 映射到了同一索引上。
  2. 哈希表里面已经装了太多 key - value 了，这种情况下，哈希函数再完美，也没办法避免哈希冲突。

针对第二种情况，引出了负载因子的概念

负载因子是一个哈希表装满的程度的度量。一般来说，负载因子越大，说明哈希表里面存储的键值对越多，哈希冲突的概率越大，哈希表的操作性能越差。

负载因子计算公式：
$$\frac{size}{table.size()}$$
其中，size 就是哈希表里面的现有键值对数量，table.size() 是哈希表底层数组的容量。

不难发现，用拉链法实现的哈希表，负载因子可以无限大，因为链表可以无限延伸；用线性探测法实现的哈希表，负载因子不会超过1。

当哈希表内元素达到负载因子阈值时，哈希表就会发生扩容操作，会将哈希表底层 table 数组的容量扩大，把数据重新计算哈希值搬移到新的大数组中，size 不变， 容量增加，负载因子就减小了。
