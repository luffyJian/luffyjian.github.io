哈希表的种种变换

发表于2025-11-14|更新于2025-11-17
|总字数:2.2k|阅读时长:11分钟|浏览量:

用链表加强哈希表

从哈希表原理分析知道，不能依赖哈希表遍历 key 的顺序，即哈希表中的 key 是无序的。

如果需要按照插入时顺序遍历 key，可以用链表加强哈希表。

这样的哈希链表：

  • 可以在 $O(1)$ 的时间复杂度查找到对应的双链表节点，进而找到键的对应的值。
  • 可以在 $O(1)$ 的时间复杂度插入新的键值对，因为哈希表本身的插入操作时间复杂度是 $O(1)$，且双链表头尾插入操作的时间复杂度也是 $O(1)$。
  • 可以在 $O(1)$ 的时间复杂度删除一个键值对， 因为哈希表本身的删除操作时间复杂度是 $O(1)$，且双链表删除给定节点操作的时间复杂度也是 $O(1)$。
  • 由于链表节点的顺序是插入顺序，那么只要从头结点开始遍历这个链表，就能按照插入顺序访问所有键。

也就是说，能在不改变标准哈希表的基本操作复杂度的前提下，实现了按照插入顺序访问所有键的需求。

用数组加强哈希表

引入：基于标准哈希表的API上，再添加一个新的 randomKey() API，可以在 $O(1)$ 的时间复杂度随机返回一个键。

注意，我们说的随机，都是指均匀随机，即每个元素选中的概率相等。

在标准数组中，只要用一个随机数生成器，随机生成一个索引，就相当于找到了一个随机元素。这有个前提，就是这个数组中间没有空洞元素，元素都是连续的。

哈希表的底层 table 数组不仅包含空洞元素，情况还可能更加复杂一些

如果是开放寻址法，就是带空洞数组的场景，如果是链地址法，光随机一个索引是不够的，还得随机链表中的一个节点。

唯一的办法就是通过 keys 方法遍历整个 table 数组， 把所有的键存储到一个数组中，然后再随机返回一个键。但这样时间复杂度为 $O(N)$，不符合要求。

使用数组来加强哈希表来实现 randomKey() API。
